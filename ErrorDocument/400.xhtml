Bad Request

Your Web browser sent a request that this Web server didn't understand. Most likely, this is because of a bug in the way in which your Web browser sends SNI host names. This bug is known to affect, at a minimum, the following Web browsers:

  • Chromium (Chrome, et cetera)

Most Web browsers are broken.

Malformed SNI host names

Many Web browsers exhibit a bug of one form or another when dealing with URIs that have a trailing dot in their host component. First, some background. To quote a couple specifications:

RFC 6066 (SNI) says: 

"HostName" contains the fully qualified DNS hostname of the server,
as understood by the client.  The hostname is represented as a byte
string using ASCII encoding without a trailing dot.

While RFC 7230 (HTTP) has this to say: 

A client MUST send a Host header field in all HTTP/1.1 request
messages.  If the target URI includes an authority component, then a
client MUST send a field-value for Host that is identical to that
authority component, excluding any userinfo subcomponent and its "@"
delimiter (Section 2.7.1).

On that second one, the emphasis is in the original document. I didn't add it.

If you pay attention to what these documents are saying, the SNI host name and HTTP Host header do not always match. The SNI host name must never have a trailing dot, but the HTTP Host header must reflect a host name that is identical to the host name of the URI, so if the URI's host has a trailing dot, the HTTP Host header must include that trailing dot. For example, if the URI of a page is https://alice.sni.velox.ch./, the following values should be sent by the Web browser:

SNI host: alice.sni.velox.ch

HTTP host: alice.sni.velox.ch.

While I agree that requiring that the SNI host name have its trailing dot stripped is kind of dumb, the fact is, some servers conform to this. If you send an Apache Web server a request with an invalid SNI host name that is invalid only because it contains a trailing dot, Apache will throw a 400 error at you. You can't really argue with the Apache developers for setting this up, either. After all, they followed the standard to the letter. However, most Web clients seem to have a bug that causes them to fail to trim off this trailing dot from the SNI host name. This limits the practicality of using "https"-scheme URIs. These URIs are perfectly valid, but Web clients don't handle them correctly. In fact, you could even say that these dotted domain URIs are *more* valid than their non-dotted counterparts. RFC 7230 (HTTP) makes no comments on trailing dots, and in fact, doesn't even require that the host component of the URI be a DNS name: 

If host is a registered name, the registered name is an
indirect identifier for use with a name resolution service, such as
DNS, to find an address for that origin server.

"Such as DNS". DNS isn't a requirement. Most of us do use DNS to make sense of "https"-scheme URIs though. RFC 3986 (URIs) has this to say about using a DNS domain name in a URI: 

A host identified by a registered name is a sequence of characters
usually intended for lookup within a locally defined host or service
name registry, though the URI's scheme-specific semantics may require
that a specific registry (or fixed name table) be used instead.  The
most common name registry mechanism is the Domain Name System (DNS).
A registered name intended for lookup in the DNS uses the syntax
defined in Section 3.5 of [RFC1034] and Section 2.1 of [RFC1123].
Such a name consists of a sequence of domain labels separated by ".",
each domain label starting and ending with an alphanumeric character
and possibly also containing "-" characters.  The rightmost domain
label of a fully qualified domain name in DNS may be followed by a
single "." and should be if it is necessary to distinguish between
the complete domain name and some local domain.

The dot should be included when there is a chance that not including the dot results in ambiguity with some local domain. You don't know the setup of your users. You don't know if they have local domains defined. There is *always* a chance of this ambiguity. Furthermore, RFC 7230 (HTTP) makes a comment about DNS returning different results to different users: 

When a registered name is used in the authority component, the "http"
URI scheme (Section 2.7.1) relies on the user's local name resolution
service to determine where it can find authoritative responses.  This
means that any attack on a user's network host table, cached names,
or name resolution libraries becomes an avenue for attack on
establishing authority.  Likewise, the user's choice of server for
Domain Name Service (DNS), and the hierarchy of servers from which it
obtains resolution results, could impact the authenticity of address
mappings; DNS Security Extensions (DNSSEC, [RFC4033]) are one way to
improve authenticity.

The user's choice of server and the hierarchy of servers can impact the result. Admittedly, upon first glance, I thought that it was talking about the DNS name hierarchy, which would have directly validated my point. However, while that's not exactly what is said, it has a similar impact. Depending on which server you use, DNS results can come out differently. One of the main and completely standard reasons that a DNS server (or for that matter, any DNS server in a chain of DNS servers) might return different results than another, is that one or both DNS servers is resolving the name within a local domain name space. The obvious way to insure that the domain is resolved in a global context is to include the trailing dot, and as a result, it is helpful for clients to be able to correctly handle URIs that include the trailing dot.

Web clients that exhibit this bug:

Arora

The Arora developer believes that the bug is in the underlying QT networking library, not in Arora itself. A bug report has been filed with the Qt developers, and a bug fix has been written and accepted into the version 5.6 code. It seems that version 5.6 has already been released though, so it's possible that the Gerrit branch won't be renamed until version 5.7 is ready to be released. At this point, all we need to do is wait for the next Qt release, then wait for the new release to propagate to the software distributions that include Qt.

Chromium (Chrome, et cetera)

Reporting bugs to the Chromium developers requires a Google account, and getting a Google account requires a telephone number. This is moronic. While I wasn't able to report the bug myself, I did find someone that was willing to file a bug report on my behalf. It turns out though that is was already a known bug. The bug has been known about for almost a year, but has not yet been fixed.

Epiphany

I filed a bug report with the GNOME developers and it has been acknowledged. It appears that the bug is believed to be specifically in the glib code.

Firefox (Iceweasel, Tor Browser Bundle, Pale Moon, et cetera)

I filed a bug report with Mozilla, but it turns out that this was a known bug. However, the patch included was from cURL, which as you can read below, introduces a different bug.

Midori

I filed a bug report with the Midori developers, but they have not yet responded.

Konqueror

I filed a bug report with the KDE developers. It has not yet been acknowledged, but if it turns out that the issue relates to the underlying Qt networking library like it likely does in Arora, the may resolve itself once the Qt networking library is rereleased and updated in projects that depend on it.

Surf

Suckless, the developers of Surf, believe that the error isn't in their own code, but in Webkit, so I filed a bug report with the Webkit developers. w3m

Getting an account on SourceForge is a bit of a pain, due to IP address sniffing on the registration page. I've filed a bug report, but I somehow get the feeling that this project is unmaintained. The last update seems to have been on 2011-01-15. If I'm right, a bug fix may not come any time soon. If you use w3m, it might be time to switch to a Web browser that still has support.

Wget

I filed a bug report with the FSF. Most of the conversation occurred via email though, and I don't know how to find the relevant bits in the online archive. Long story short, is sounds like if they decide to fix this bug, they will add a new bug, the one from cURL.

Incorrect HTTP Host names

Sending malformed SNI host names isn't the only mis-processing of URIs by Web browsers that seems to be triggered by fully-qualified domain names. Some Web browsers send the correct SNI Host names, but they strip the trailing dot off of not only the SNI host name, but also the HTTP Host header. The HTTP Host header is used by the Web server to know what site is being requested, and as mentioned above, this header is *required* to contain the host name as it appears in the URI. The site that uses the trailing dot host name is not necessarily the same site that uses the host name without the trailing dot. The server administrator can choose to host two different sites using these host names, similar to how a server administrator could host another site using a subdomain. Furthermore, a server administrator may host a single website, but have a preferred host name. This is like when a "www.*" website redirects to a non-"www.*" website or vice versa.

I've seen some websites that redirect from a trailing-dot domain to a domain without the trailing dot. These websites have every right to choose the version of the domain without the trailing dot as their canonical version. However, going back to the possibility of domains being resolved in a local name space, redirecting from the domain //example.net. (with a trailing dot) to //example.net (without a trailing dot) could result in the latter being resolved to //example.net.example.org., causing the user to be unable to reach //example.net.. This is a very unlikely case, but it still means that some server administrators would choose the trailing-dot version of their domain as the canonical host. When Web clients fail to transmit the correct Host header, this will cause an endless redirect loop. The client obviously stops attempting to retrieve the page after a while, but the user is left unable to reach the content that they were after. Again, if you read the standards-setting RFCs, this is the fault of the client software, not the server and not the server's administrator.

Other strange behavior

I found a singular strange case when trying to find which Web clients have the SNI bug and which don't. Namely, the Netsurf Web browser normalizes URIs that have a trailing dot to their form without the trailing dot. Nowhere in any standards-setting documentation can I find any support for this behaviour. However, as stated above, "https"- and "http"-scheme URIs are not required to be resolved using DNS. Effectively, Netsurf is resolving URIs using an internal non-DNS mechanism. Names with a trailing dot are then redirected to their version without the trailing dot, while names without the trailing dot are sent to the DNS resolver. While this behaviour has the same problems as simply stripping the dot off of the Host header, I have no foothold with which to call this a bug. It's certainly undesirable behaviour for my use case, but my best guess is that the standards allow it. Perhaps when I have time, I'll report this issue as a feature request.

Conclusions

I've stated above that I feel that the trailing-dot version of a domain seems more correct to me than the version without a trailing dot. However, please don't take that to mean that I think all websites and/or URI should make use of the trailing dot. Really, it's a matter of choice, and both options should be available. All I want is for Web clients to follow the standards so that such a choice is possible. My point about the trailing-dot version being more valid is that if one version of the domain is to be ruled out, it makes far more sense to require the trailing dot than to disallow it.
